## MySQL的逻辑架构

总体来说，MySQL可以分为两部分：Server层和存储引擎层。

其中：

- Server层，包括连接器、查询缓存、分析器、优化器、执行器等，它涵盖了MySQL的大多数核心服务功能，以及所有的内置函数（如日期、时间、数学和加密函数等）。此外，所有的跨存储引擎功能都在这一层实现，比如，存储过程、触发器、视图等。
- 存储引擎层，负责数据的存储和提取。它以插件的形式存在，支持InnoDB、MyISAM、Memory等多个存储引擎。现在最常用的存储引擎是InnoDB，它从MySQL 5.5.5版本开始成为了默认的存储引擎。



## 连接器 - 连接管理、权限验证

### 简介

连接器负责跟客户端建立连接、获取权限、维持和管理连接。

在完成了经典的TCP三次握手后，连接器就会开始进行身份认证。此时，就会用到我们输入的用户名和密码，如果用户名或密码不对，我们就会收到一个“Access denied for user”的错误；而如果用户名和密码通过认证，连接器就会到权限表中查询出用户所拥有的权限。之后，这个连接所涉及到的权限判断，都依赖于此时读到的权限。

这也就意味着，当一个用户成功建立连接后，即使我们用管理员账号对这个用户的权限做了修改，也不会影响已经存在的连接的权限。修改完成后，只有再新建的连接才会使用新的权限设置。



成功建立一个连接后，如果我们没有任何动作，这个连接就处于空闲状态。我们可以通过show processlist命令看到它：

其中，Command列显示为“sleep”，就表示这是一个空闲的连接。

而当客户端长时间没有动静时，连接器就会自动将它断开。这个时间是由参数wait_timeout控制的，默认值是8小时。在这个连接被断开之后，如果客户端再次发送请求的话，就会收到一个错误提醒：Lost connection to MySQL server during query。这时，如果我们要继续，就需要重连，然后再次执行请求。



### 长连接

在数据库中，长连接是指连接成功后，如果客户端持续有请求，则一直使用同一个连接。短连接则是指每次执行完很少的几次查询就断开连接，下次查询则重新建立一个。

建立连接的过程通常是比较复杂的，也是比较昂贵的，因此，在使用中我们要尽量减少连接的建立，也就是要尽量使用长连接。

但是，全部使用长连接后，我们会发现，有些时候MySQL占用的内存空间，会涨得特别快，这是因为MySQL在执行过程中临时使用的内存是管理在连接对象里面的，而这些资源只有在连接断开时才会被释放。所以，如果长连接累积下来，就可能会导致内存占用过大，被系统强行杀掉（OOM），从现象上看就是MySQL异常重启了。



> 那么，我们该怎么解决这个问题呢？

我们可以考虑以下两种方案：

- 定期断开长连接。在使用一段时间后、或者通过程序判断执行过一个占用内存的大查询后，就断开连接，之后要查询则重连。
- 借助mysql_reset_connection的特性。在MySQL 5.7或更新的版本中，可以在每次执行一个大的操作后，通过执行mysql_reset_connection来重新初始化连接资源。这个过程不需要重连和重新做权限验证，但是会将连接恢复到刚刚创建完时的状态。



## 查询缓存

### 简介

当MySQL拿到一个查询请求后，首先会到查询缓存中看看，之前是不是执行过这条语句。之前执行过的语句及其结果可能会以key-value的形式，被直接缓存在内存中。其中，key是查询的语句，value是查询的结果。

如果我们的查询能够直接在这个缓存中找到对应的key，那么这个value就会被直接返回给客户端。而如果语句不在查询缓存中，就会继续后面的执行阶段。执行完成后，执行结果会被存入查询缓存中。

可以看到，如果查询命中缓存，MySQL就不需要执行后面的复杂操作，就可以直接返回结果，这个效率会很高。



### 为什么不建议使用查询缓存？

> 但是，在大多数情况下我们并不建议使用查询缓存，这是为什么呢？

因为查询缓存往往弊大于利。查询缓存的失效往往非常频繁，只要对一个表进行更新，这个表上所有的查询缓存都会被清空。因此，很可能我们费劲存起来的结果，往往还没到使用，它就被一个更新清空了。

对于更新压力很大的数据库来说，查询缓存的命中率会非常低。除非我们的业务就是一张静态表，很长时间才会更新一次。比如，一个系统配置表，那这张表上的查询才适合使用查询缓存。



### 按需使用查询缓存

对于查询缓存，MySQL提供了“按需使用”的方式。我们可以将参数query_cache_type设置成DEMAND，这样对于默认的SQL语句都不会使用查询缓存。而对于那些确定要使用查询缓存的语句，可以用SQL_CACHE显示指定，如下：

```mysql
select SQL_CACHE * from T where ID=10;
```

需要注意的是，在MySQL 8.0中直接将查询缓存的整块功能删掉了，也就是说从 8.0 开始彻底没有这个功能了。



## 分析器 -  词法分析、语法分析

当我们的查询没有命中缓存时，就会开始真正执行语句了。首先，MySQL需要知道我们要做什么，因此，它会对SQL语句做解析。在分析SQL语句时：

- 首先，分析器会做“词法分析”。我们的输入通常是由多个字符串和空格组成的，MySQL需要识别出里面的字符串分别是什么、代表什么。比如，MySQL会从我们输入的“select”这个关键字，识别出这是一个查询语句。

- 然后，分析器会做“语法分析”。根据词法分析的结结果，语法分析器会根据语法规则，判断我们输入的SQL语句是否符合MySQL语法。



当我们输入的语句不对时，就会收到`You have an error in your SQL syntax`的错误提醒。比如，在下面这个语句中select少打了开头的字母”s“：

```mysql
elect * from t where ID=1;
```

此时，我们会收到如下错误：

```mysql
ERROR 1064 (42000): You have an error in your SQL syntax; check the manual that corresponds to your MySQL server version for the right syntax to use near 'elect * from t where ID=1' at line 1
```

注：对于语法错误，通常会提示第一个出现错误的位置，所以我们要关注的是紧接”use near“的内容。



## 优化器 - 生成执行计划（选择索引、确定连接顺序）

经过分析器的分析之后，MySQL就知道我们要做什么了。但在实际执行之前，还要经过优化器的处理。



优化器，用于在表中有多个索引时，决定使用哪个索引；或者在一个语句有多表关联时，决定各个表的连接顺序。比如，我们要执行的语句是这样的：

```mysql
select * from t1 join t2 using(ID)  where t1.c=10 and t2.d=20;
```

可以看到，这个语句是执行两个表的join：

- 即可以先从t1里面取出c=10的记录的ID值，再根据ID值关联到表t2，再判断t2里面d的值是否等于20。
- 也可以先从t2里面去除d=20的记录的ID值，再根据ID值关联到表t1，再判断t1里面c的值是否等于10。

这两种执行方法的逻辑结果是一样的，但是执行的效率会有不同，而优化器的作用就是决定选择使用哪一个方案。

优化器阶段完成后，语句的执行方案就确定下来了，然后会进入执行器阶段。



## 执行器 - 操作引擎，返回结果集

MySQL通过分析器知道了我们要做什么，通过优化器知道了该如何做，然后就可以进入执行阶段，开始执行语句了。



在执行开始时，执行器会先判断我们对这个表没有执行查询的权限。如果没有，就会返回没有权限的错误，如下所示：

```
mysql> select * from T where ID=10;

ERROR 1142 (42000): SELECT command denied to user 'b'@'localhost' for table 'T'
```

如果有权限，就会打开表继续执行。在打开表时，执行器就会根据表的引擎定义，使用这个引擎提供的接口。

> 注：在工程实现上，如果命中查询缓存，就会在查询缓存返回结果的时候，做权限验证。



我们假设，这个例子中的表T，在ID字段上没有索引，那么执行器的执行流程是这样的：

- 调用InnoDB引擎接口取这个表的第一行，判断ID值是不是10，如果不是则跳过，如果是则将这行存在结果集中；
- 调用引擎接口取”下一行“，重复相同的判断逻辑，直到取到这个表的最后一行。

执行器会将上述遍历过程中所有满足条件的行组成的记录集作为结果集返回给客户端。这样，这个语句就执行完成了。

对于有索引的表，执行的逻辑也差不多。第一次调用的是”取满足条件的第一行“这个接口，之后循环取”满足条件的下一行“这个接口，这些接口都是引擎中已经定义好的。



在数据库的慢查询日志中，我们可以看到一个rows_examined字段，它表示这个语句在执行过程中扫描了多少行。而这个值是在执行器每次调用引擎获取数据行时累加的。

需要注意的是，有些场景下，执行器调用一次引擎，在引擎内部可能会扫描多行，因此引擎扫描的实际行数跟rows_examinde并不是完全相同的。
