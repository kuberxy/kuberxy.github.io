## 事务简介

### 什么是事务

简单来说，事务就是一组数据库操作的集合。一个事务可以是一条简单的查询语句，也可以是多条增删改查语句。事务的最大特点是，它能保证这组操作，要么全部成功，要么全部失败。

对于事务，最经典的例子就是转账，比如你要给朋友转100块钱。通常一个转账过程，在具体的程序实现中会涉及一系列的操作。比如，查询余额、做加减法、更新余额等。而这些操作必须保证是一体的，不然在对你的账户做了减法之后，还没在朋友的账户做加法之前，程序或数据库异常，导致这一系列操作被中断了。此时你和朋友的帐就对不上了。如果这样，银行系统不就乱套了吗？因此，这时就必须要用事务了。



在MySQL中，对事务的支持是在引擎层实现的。我们知道，MySQL是一个多引擎的系统，但不是所有的引擎都支持事务。比如，MySQL原生的MyISAM引擎就不支持事务，这也是它被InnoDB取代的重要原因之一。

而一提到事务，熟悉数据库的朋友肯定对想到ACID（Atomicity、Consistency、IsoIation、Durability，即原子性、一致性、隔离性、持久性）。今天我们就来说说其中 I，也就是“隔离性”。



### 事务的启动方式

在MySQL中，我们通常会使用如下两种方式来启动一个事务：

- 设置autocommit=1开启线程的自动提交功能，并使用显示的事务启动语句。比如，begin或start transaction，而配套的提交语句是commit，回滚语句是rollback。
- 设置autocommit=0关闭线程的自动提交功能。这样当我们执行一个select语句时，事务就启动了，并且不会自动提交。同时，这个事务会持续存在直到主动执行commit或rollback语句，或者断开连接。

通常，建议设置autocommit=1，并使用显示的事务启动语句。该方法会存在一个”多一次交互“的问题。即对于一个需要频繁使用事务的程序来说，该方法在每个事务开始时都需要主动执行一次”begin“，增加了语句交互的次数。此时，我们可以使用commit work and chain的语法。

在autocommit为1的情况，对于通过begin显示启动的事务，当执行commit时，则提交事务。当执行commit work and chain时，则是提交事务并自动启动下一个事务，这样就省去了再次执行begin语句的开销。同时带来的好处是从程序开发的角度明确地知道每个语句是否处于事务中。



## 隔离级别

当数据库上存在并发时，即有多个事务在同时执行，此时就可能出现藏读（dirty read）、幻读（phantom read）、不可重复读（non-repeatable read）的问题。为了解决这些问题，就有了“隔离级别”的概念。

>  我们需要知道的是，隔离得越严实，效率就会越低。因此，我们需要在二者之间寻找一个平衡点。



### 四种隔离级别

SQL标准的事务隔离级别包括：

- 读未提交（read uncommitted），是指一个事务还没提交时，它做的变更就能被其他事务看到。

- 读提交（read committed），是指一个事务提交之后，它做的变更才能被其他事物看到。

- 可重复读（repeatable read），是指一个事务在执行过程看到的数据，总是跟这个事务在启动时看到的数据是一致的。当然，在该级别下，一个事务未提交的变更对其他事务来说，也是不可见的。

- 串行化（serializable），是指对于同一行记录，写会加写锁，读会加读锁。当出现读写锁冲突时，后访问的事务必须等前一个事务执行完成，才能继续执行。



### 理解四种隔离级别

在四种隔离级别中，读提交和可重复读比较难理解。下面，我们通过一个例子来理解它们。



假设数据表T中只有一列c，其中只有一行值为1的记录：

```mysql
create table T (c int) engine=InnoDB;
insert into T (c) values (1);
```

下面是按照时间顺序执行两个事务的行为：

![image-20200503155724923](/2020-05-03-MySQL-3/assets/images/image-20200503155724923.png)

现在，我们来看一下，在不同的隔离级别下，事务A会有哪些不同的返回结果，也就是图中V1、V2、V3的返回值分别是什么？



#### 读未提交

若隔离级别为“读未提交”，则V1值是2。这时事务B虽然还未提交，但结果可以被A看到。因此V2、V3的值也是2。

![image-20200503162408967](/2020-05-03-MySQL-3/assets/images/image-20200503162408967.png)



#### 读提交

若隔离级别为“读提交”。事务B的更新在提交后才能被A看到，所以，V1的值是1，V2、V3的值是2。

![image-20200503162843079](/2020-05-03-MySQL-3/assets/images/image-20200503162843079.png)



#### 可重复读

若隔离级别为“可重复读”。事务A在执行期间看到的数据和启动事务时看到的数据一致，所以V1、V2的值是1，V3的值是2。

![image-20200503163231540](/2020-05-03-MySQL-3/assets/images/image-20200503163231540.png)



#### 串行化

若隔离级别为“串行化”，则在启动事务后，A和B会先后都会对行记录加读锁。在事务B执行“将1改成2”的时候，会对行记录加写锁，此时发生了读写锁冲突，因此事务B会被锁住，直到事务A提交后，事务B才会继续执行。所以从A的角度看，V1、V2的值是1，V3的值是2。

![image-20200503164438532](/2020-05-03-MySQL-3/assets/images/image-20200503164438532.png)

### 隔离级别 & 视图

从具体的实现上，这些隔离级别其实是通过视图实现的。在数据库中会创建一个视图，访问的时候以视图的逻辑结果为准：

- 在“读未提交”隔离级别下，没有视图的概念，因此会直接返回记录上的最新值。
- 在“读提交”隔离级别下，这个视图是在每个SQL语句开始执行的时候创建的。
- 在”可重复读“隔离级别下，这个视图是在事务启动时创建的，整个事务在存在期间都用这个视图。
- 在”串行化“隔离级别下，则是直接用加锁的方式来避免并行访问的。

我们可以看到在不同隔离级别下，数据库的行为是不同的。



### 查看、配置MySQL的隔离级别

Oracle数据库的默认隔离级别为”读提交“，因此对于一些从Oracle迁移到MySQL的应用，为保证数据库隔离级别的一致，一定要记得将 MySQL的隔离级别设置为”读提交“。



配置的方式是，将启动参数 transaction-isolation 的值设置成 READ-COMMITTED。我们可以使用用 show variables 来查看当前的值。

```mysql
mysql> show variables like 'transaction_isolation';

+-----------------------+----------------+

| Variable_name | Value |

+-----------------------+----------------+

| transaction_isolation | READ-COMMITTED |

+-----------------------+----------------+
```



### 小结

总的来说，存在即合理，每种隔离级别都有自己的使用场景，我们要根据自己的业务情况来定。



> 思考：什么时候需要”可重复读“的场景？

假设，有一个个人银行账户的库。其中，一个表存了账户余额，一个表存了账单明细。到了月底我们要对账（数据校对），也就是判断上个月的余额和当前余额的差额，是否与本月的账单明细一致。此时，在校对过程中，即使有用户发生了一笔新交易，我们也不希望它影响到我们的校对结果。

这时，使用”可重复读“隔离级别就很方便了。事务启动时的视图可以认为是静态的，不会受到其它事务的更新操作的影响。



## 事务隔离的实现

在理解了事务的隔离级别后，我们再看看看事务隔离具体是怎么实现的？下面，我们以”可重复读“隔离级别来说明。



### 回滚日志

实际上，MySQL在更新一条记录时，会为这条记录生成一条回滚操作并保存在回滚日志（undo log）中。因此，记录上的最新值，通过回滚操作，可以得到之前的状态值。比如，一个值从1被按顺序改成了2、3、4，那么，在回滚日志中就会有类似下面的记录

![image-20200503181529714](/2020-05-03-MySQL-3/assets/images/image-20200503181529714.png)

我们可以看到，当前值是4，但是在查询这条记录时，不同时刻启动的事务会有不同的read-view。

如图，在视图A、B、C里面，这个记录的值分别为1、2、4，即同一条记录在系统中存在多个版本，而这就是数据库的多版本并发控制（MVCC）。对于read-view A，要得到1，就必须将当前值依次执行图中所有的回滚操作得到。

此外，我们可以得出，即使现在有另外一个事务正在将4改成5，这个事务跟read-view A、B、C对应的事务是不会冲突的。

![image-20200503182632679](/2020-05-03-MySQL-3/assets/images/image-20200503182632679.png)

> 那么回滚日志总不能一直保留吧，什么时候删除呢？

答案是，在不需要的时候才删除。也就是说，系统会判断，当没有事务需要用到这些回滚日志时，回滚日志就会被删除。

> 那什么时候，事务才不需要这些回滚日志呢？

答案是，当系统中没有比这个回滚日志更早的read-view时



### 长事务

在理解了事务隔离的实现后，我们再来讨论一下为什么建议不使用长事务？

- 在MySQL 5.5及以前的版本，回滚日志是跟数据字典一起放在ibdata文件中的，即使长事务最终被提交，回滚段（日志）被清理，文件也不会变小。因此，我们可能会遇见这样的场景，数据只有几十GB，而回滚段却又几百GB的库。此时，就只能清理会滚段，重建整个库了。

- 此外，长事务还会占用锁资源，这也可能会拖垮整个库。



> 既然长事务会有潜在的风险，那么我们应该如何避免使用长事务呢？

长事务，通常是由误用导致的。比如，有些客户端框架会在连接成功后，默认先执行一个set autocommit=0的命令。这就导致接下来的查询都在一个事务中，同时，如果这个连接是长连接，就导致了意外的长事务。

我们可以通过information_schema库中的innodb_trx表查询长事务。如下语句，用于查找持续时间操过60s的事务：

```mysql
select * from information_schema.innodb_trx where TIME_TO_SEC(timediff(now(),trx_started))>60
```
