接触过数据库的朋友，对”索引“一词肯定不会陌生。但我们能准确的回答下面的两个问题吗？

- 什么是索引？
- 索引是如何工作的？



## 什么是索引？

简单来说，索引就像书的目录，它的出现就是为了提高查询效率。比如，一本500页的书，在不借助目录的情况下，想要找到某个知识点，一定得花费大量的时间。同样，对于数据库中表来说，索引就是它的”目录“。



## 索引的常见数据结构

对于一张完整的数据表而言，索引就是一张表更加轻量的数据表，而实现这张轻量级数据表的数据结构又有很多种。比如，哈希表、有序数组和搜索树。



### 哈希表

哈希表，是一种以键 - 值（key - value）形式来存储数据的结构。我们只要输入待查找的键（key），就可以找到其对应的值（value）。

哈希的思路很简单，借助数组并通过哈希函数把key换算成一个确定的位置，然后把value放在数组的这个位置。



多个key值经过哈希函数的换算，不可避免地，会出现同一个值得情况。而处理这种情况的一种方法是，拉出一个链表。假设，我们现在维护着一个身份证信息表，需要根据身份证号查找对应的名字，此时对应的哈希索引的示意图如下：

图中，User2和User4根据身份证号算出来的值都是N，此时，它对应的是一个链表。假设，我们要查ID_card_n2对应的名字，此时的处理步骤如下：

- 首先，将ID_card_n2通过哈希函数算出N；
- 然后，按顺序遍历，找到User2。

需要注意的是，图中四个ID_card_n的值并不是递增的，这样做的好处是，增加新的User时速度会很快，只需要往后追加。但缺点是，因为不是有序的，所以哈希索引做区间查询的速度会很慢。比如，我们现在要找身份证号在[ID_card_X,ID_card_Y]这个区间的所有用户，此时，就必须扫描全部的key了。

因此，哈希表这种结构适用于只有等值查询的场景，比如，Memcached、Redis等一些NoSQL引擎。



### 有序数组

有序数组在等值查询和范围查询场景中的性能就都非常优秀了。还是以身份证号查名字为例，如果我们使用有序数组来实现，示意图如下：

这里我们假设身份证号没有重复，这个数组就是按照省份证号递增的顺序保存的。这时，如果我们要查ID_card_n2对应的名字，用二分法就可以快速得到，而此时的时间复杂度是O(log(N))。

同时，这个索引结构是支持范围查询的。我们要查身份证号在[ID_card_X, ID_card_Y]区间的User，可以先用二分法找到ID_card_X，然后向右遍历，直到查到第一个大于ID_card_Y的身份证号，最后退出循环。



如果仅看查询效率，有序数组就是最好的数据结构了。但是，在需要更新数据的时候就很麻烦了，当我们往中间插入一条记录就必须得挪动后面所有的记录，成本太高。

因此，有序数组只适用于静态存储引擎。比如，要保存的是2017年某个城市的所有人口信息，这类不会再修改的数据。



### 搜索树

我们还是以身份证号查名字为例，如果我们用二叉树来实现的话，示意图如下：

二叉树的搜索特点是：每个节点的左儿子小于父节点，父节点又小于右儿子。这样如果你要查ID_card_n2的话，按照图中的搜索顺序就是按UserA -> UserB -> UserC -> UserF -> User2这个路径得到。这个时间复杂度是O(log(N))。

> 为了维持O(log(N))的查询复杂度，就需要保持这颗树是平衡二叉树。为了做这个保证， 更新的时间复杂度也是O(log(N))

而树可以是二叉，也可以是多叉。多叉树就是每个节点有多个儿子，儿子之间的大小是从左到右递增的。二叉树是搜索效率最高的，但是实际上大多数的数据库存储却并不使用二叉树。其原因是，索引不止存在内存中，还要写到磁盘上。

我们可以想象一下，一颗100万节点的平衡二叉树，树高20。那么，一次查询就可能需要访问20个数据块。在机械硬盘时代，从磁盘随机读一个数据块需要10ms左右的寻址时间。也就是说，对于一个100万行的表，如果使用二叉树来存储，单独访问一行可能需要200ms的时间。

为了让一个查询尽量少地读磁盘，就必须让查询过程访问尽量少的数据块。因此，就不应该使用二叉树了， 而是要使用”N叉“树。这里的N取决于数据块的大小。

以InnoDB的一个整数字段索引为例，这个N差不多是1200。当这棵树高是4的时候，就可以存1200的3此方，即17亿。此外，考虑到树根的数据块总是在内存中的，对于一个10亿行的表来说，一个整数字段的索引，查找一个值最多只需要访问3次磁盘。而且，树的第二层大概率也是在内存中的，那么访问磁盘的平均次数就更少了。



N叉树由于在读写上的性能优点，以及适配磁盘的访问模式，已经被广泛应用在数据库引擎中了。



## InnoDB的索引模型

在MySQL中，索引是在存储引擎层实现的。因此，并没有统一的索引标准，即不同存储引擎的索引的工作方式并不一样。即使多个存储引擎支持同一种类型的索引，其底层的实现也可能不同。

在InnoDB引擎中，表都是根据主键顺序以索引的形式存放的。而采用这种存储方式的表就被称为索引组织表。我们知道，InnoDB使用了B+树索引模型，所以数据都是存储在B+树中的。

> 在InnoDB中，每一个索引对应着一棵B+树



假设，我们有一个主键列为ID的表，表中有字段k，并且在k上有索引。这个表的建表语句如下：

```mysql
create table T(
id int primary key, 
k int not null, 
name varchar(16),
index (k))engine=InnoDB;
```

表中 R1~R5 的 (ID,k) 值分别为 (100,1)、(200,2)、(300,3)、(500,5) 和 (600,6)。此时，两棵树的示例示意图如下：

可以看出，根据叶子节点的内容，索引类型分为主键索引和非主键索引：

- 主键索引的叶子节点存的是整行数据。在InnoDB中，主键索引也被称为聚簇索引（clustered index）。
- 非主键索引的叶子节点存的是主键的值。在InnoDB中，非主键索引也被称为二级索引（secondary index）。

而基于主键索引和普通索引的查询也是有区别的：

- 如果语句是 select * from T where ID=500，即主键索引查询，只需要搜索ID这棵B+树；
- 如果语句是 select * from T where k=5，即普通索引查询，则需要先搜索k索引树，得到ID的值为500，再到ID索引树搜索一次。这个过程就被称为回表。

也就是说，基于非主键索引的查询需要多扫描一棵索引树。因此，我们应该尽量使用主键查询。



## 索引维护

### 页分裂 & 页合并

B+树为了维护索引的有序性，在插入新值时需要做必要的维护。

我们还是以上图为例：

如果要插入的新行的ID值为700，则只需要在R5记录后追加这条新记录。而如果要插入的新行的ID值为400，就需要逻辑上挪动后面的数据，空出位置。

而更糟的情况是，如果R5所在的数据页已经满了，根据B+树算法，这时就需要申请一个新的数据页，然后挪动部分数据过去。这个过程就被称为也分裂。在这种情况下，性能自然会受影响。

除了性能外，页分裂操作还影响数据页的利用率。原本放在一个页的数据，现在分到两个页中，整体空间利用率降低大约50%。

当然，有分裂就会有合并。当相邻两个数据页由于删除了数据，利用率很低之后，就会将数据页做合并。合并的过程，可以认为是分裂过程的逆过程。



### 案例：自增主键的使用

在一些建表规范中，我们一定见过这样的描述：建表语句中一定要有自增主键。但这是条建议”放之四海而皆准“吗？

自增主键，是指在自增列上定义的主键。在建表语句中一般是这么定义的：

```mysql
not null primary key auto_increment
```

使用了自增主键后，在插入新记录的时候，可以不指定ID的值。此时，系统会获取当前ID最大值加1作为下一条记录的ID值。

简单来说，自增主键的数据插入模式是一种递增插入的场景。即每次插入一条新纪录，都是追加操作，都不涉及到挪动其它记录，也不会触发叶子节点的分裂。

然而，用有业务逻辑的字段做主键，则往往不容易保证有序插入，这样写数据的成本相对比较高。



除了考虑性能外，我们还可以从存储空间的角度来看。假设，在表中确实有一个唯一字段，比如，字符串类型的身份证号。此时，应该用身份证号做主键，还是用自增字段做主键？

因为在每个非主键索引的叶子节点上都是主键的值。如果用身份证号做主键，那么每个非主键索引的叶子节点占用约20个字节，而如果用整型做主键，则只要4个字节，如果是长整型（bigint）则是8个字节。

显然，主键长度越小，普通索引的叶子节点就越小，普通索引占用的空间也就越小。



因此，从性能和存储空间方面考量，自增主键往往是更合理的选择。



> 那在什么场景下，适合直接用业务字段做主键呢？

比如，一些业务场景的需求是这样的：

- 只有一个索引
- 该索引必须是唯一的

可以看出，这就是典型的KV场景。由于没有其它索引，所以也就不同考虑其它索引的叶子节点大小问题。此时，我们就要优先考虑”尽量使用主键查询“的原则了。
