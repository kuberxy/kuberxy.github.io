## 事务 & 锁

我们知道，在可隔离级别下，事务启动时会创建一个read-view，之后该事务在执行期间，即使有其它事务修改了数据，该事务看到的数据仍然跟它在启动时看到的一样。这也就是说，在可重复读隔离级别下，执行的事务不受外界的影响。

此外，我们还知道，当一个事务要更新一行记录时，如果刚好有另外一个事务拥有这一行的行锁，那么该事务就会被锁住，进入等待状态。

此时，问题就来了，既然事务被锁住进入了等待状态，那么等到这个事务获取到行锁要更新数据的时候，它读到的值又是什么呢？



我们可以通过下面的例子来理解这个场景，假设有一个只有两行记录的表，其定义如下：

```mysql
CREATE TABLE `t` (
  `id` int(11) NOT NULL,
  `k` int(11) DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB;

insert into t(id, k) values(1,1),(2,2);
```

现在有三个事务（A、B、C），它们的执行流程如下（注：这里默认autocommit = 1）：

![image-20200504105114175](/2020-05-03-MySQL-4/assets/images/image-20200504105114175.png)

如上，在三个事务中，事务A是一个只读事务，并且在时间顺序上它在事务B的查询之后；事务B是一个读写事务，在更新了行之后，立即进行了查询；事务C没有显示式地使用begin/commit，表示这个update语句本身就是一个事务。

需要注意的是，begin/start transaction命令并不是一个事务的起点，在执行了它们之后，第一个操作InnoDB表的语句，才是事务真正的开始。如果想要马上启动一个事务，可以使用start transaction with consistent snapshot这个命令。对于这两种启动方式来说，它们创建一致性视图的时机是不同的：

- 第一种启动方式：一致性视图在执行第一个快照读语句时创建
- 第二种启动方式：一致性视图在执行start transaction with consistent snapshot时创建



> 这里，我们的问题是，事务A和事务B的查询结果是多少？

答案是，事务A查到的k是1，事务B查到的k是3。

看到这个结果你是否和一样晕菜了呢？为什么事务A查到结果是1。接下来，让我们一起来理解一下它们吧。



## MySQL快照

### 视图

在理解“快照”之前，我们首先要了解MySQL中的“视图”。在MySQL中，有两个“视图”的概念：

- 一个是view。它是一个用查询语句定义的虚拟表，在调用它时，才会执行查询语句并生成结果。
- 一个是InnoDB在实现MVCC时用到的一致性读视图，即consistent read view，它用于支持读提交（RC，即Read Commited）和可重复读（RR，即Repeatable Read）隔离级别的实现。

它们都没有物理结构，作用是在事务执行期间用来定义“我们能看到什么样的数据”。而MySQL的快照就依赖于这个一致性读视图。



### 快照是怎么实现的？

在可重复读隔离级别下，事务在启动时就“拍了个快照”。而且这个快照是基于整库的。

这时，经常使用数据库的朋友可能会产生疑问了，如果是基于整库快照的，那么当数据量很大时，快照创建的过程应该会有一定的时间消耗，为什么我平时使用时完全感知不到呢？

而要消除这个疑问，就必须理解快照是怎么实现的。



#### 事务ID & 行记录的多版本

在InnoDB中，每个事务都有一个唯一的transaction id，即事务ID。它是在事务启动时，向InnoDB的事务系统申请的，是按申请顺序严格递增的。

而在InnoDB表中，每行数据都是有多个版本的。每次事务更新数据时，都会生成一个新的数据版本，并且会把自己的transaction id赋值给这个数据版本，记为row trx_id。此外，旧的数据版本会被保留，并且在新的数据版本中，有信息可以直接拿到它们。

简单来说，就是数据表中的一行记录，可能存在多个版本（row），每个版本都有自己的row trx_id（对应一个唯一的事务）。如下，就是一个记录被多个事务连续更新后的状态

![image-20200504123856673](/2020-05-03-MySQL-4/assets/images/image-20200504123856673.png)

如图，虚线框中就是同一行数据的4个版本，当前最新版本是V4，k的值是22，它是被事务C即transaction id为25的事务等新的，因此它的row trx_id也是25。

> 我们知道，MySQL会为更新语句生成回滚日志（undo log），那此时undo log在哪呢？

实际上，图中的三个虚线箭头，就是undo log；而V1、V2、V3并不是物理上真实存在的，而是每次需要时根据当前最新版本和undo log计算出来的。比如，需要V2的时候，就是通过V4依次执行U3、U2算出来的。



#### 快照的实现

在理解了事务ID和行记录的多版本后，接下来我们开始探讨“InnoDB是怎么创建快照的”。



按照可重复读的定义，一个事务启动的时候，能够看到所有已经提交的事务结果。但是之后，这个事务执行期间，其它事务的更新对它不可见。因此，一个事务只需要在启动的时候声明说：

- 以我启动的时刻为准，如果一个数据版本是在我启动之前生成的，就认；
- 如果是在我启动之后才生成的，我就不认，我必须要找到它的上一个版本；如果“上一个版本”也不可见，那就得继续往前找。
- 而如果数据是这个事务自己更新的，那它自己还是要认的。



在具体的实现上，InnoDB为每个事务构造了一个数组，用来保存这个事务在启动瞬间，当前正“活跃”的所有事务ID。“活跃”指的就是，启动了但还没有提交。

在这个数组中，事务ID的最小值记为低水位，当前系统中已经创建过的事务ID的最大值加1记为高水位。

这个数组和高水位，就组成了当前事务的一致性视图（read-view）。



而数据版本的可见性规则，就是基于数据的row trx_id和这个一致性视图的对比结果得到的。

这个视图数组把所有的row trx_id分成了几种不同的情况。

![image-20200504142627418](/2020-05-03-MySQL-4/assets/images/image-20200504142627418.png)

这样，对于当前事务的启动瞬间来说，一个数据版本的row trx_id，有以下几种可能：

- 如果落在绿色部分，表示这个版本是已提交的事务或者是当前事务自己生成的，这个数据是可见的；
- 如果落在红色部分，表示这个版本是由将来启动的事务生成的，肯定是不可见的；
- 如果落在黄色部分，就会有如下两种情况：
  1. 若row trx_id在数组中，表示这个版本是由还没提交的事务生成的，因此，不可见；
  2. 若row trx_id不在数组中，表示这个版本是已经提交了的事务生成的，因此，可见；

比如，就前面的多版本数据来说，如果有一个事务，它的低水位是18，那么当它访问这一行数据时，就会从V4通过U3计算出V3，所以在它看来，这一行的值是11。

这样，有了这个声明之后，系统里随后发生的更新，就跟这个事务在启动时看到的内容无关了。因为之后的更新，生成的版本一定属于上面的2或者3（1）的情况，而对于该事务来说，这些新的数据版本是不存在的，所以这个事务的快照，就是“静态”的。



而InnoDB就是利用“所有数据都有多个版本”的特性，实现了“秒级创建快照”的能力。因此，快照就是启动事务时的数组 + 行记录的多版本，而行记录的多版本是通过undo log计算得来的，那也就是说，创建快照就是在启动事务时创建一个数组。



### 理解开篇的场景

好了，在理解了快照的实现后，我们继续来看在开篇的场景中所描述的三个事务，分析一下事务A查到的结果为什么是k=1？



这里，我们不妨做如下假设：

- 事务A开始前，系统里面只有一个活跃的事务，它的事务ID是99；
- 事务A、B、C的事务ID分别是100、101、102，且当前系统中只有这四个事务；
- 三个事务开始前，（1，1）这一行数据的row trx_id是90。

这样，事务A的视图数组就是[99,100]，事务B的视图数据就是[99,100,101]，事务C的视图数组就是[99,100,101,102]。因此，我们就能画出如下查询逻辑图：

![image-20200504152736186](/2020-05-03-MySQL-4/assets/images/image-20200504152736186.png)

从图中我们可以看到：

- 第一个有效更新是事务C，它把数据从(1,1)改成了(1,2)。这时候，这个数据的最新版本的row trx_id是102，而90这个版本已经成为了历史版本。
- 第二个有效更新是事务B，它把数据从(1,2)改成了(1,3)。这时候，这个数据的最新版本的row trx_id是101，而102又成为了历史版本。

我们还可以看到，在事务A查询的时候，其实事务B还没有提交，但它生成的(1,3)这个版本已经变成了当前版本了。但这个版本对事务A必须是不可见的，否则就变成了脏读。

而当事务A来读数据时，它的视图数组是[99,100]。然后，读数据都是从当前版本读起的。所以，事务A中查询语句的读流程是这样的：

- 找到(1,3)时，判断出row trx_id = 101，比高水位线大，处于红色区域，不可见；
- 接着，找到上一个历史版本，一看row trx_id = 102，比高水位线大，处于红色区域，不可见；
- 紧接着，再往前找，找到了(1,1)，它的row trx_id = 90，比低水位线小，处于绿色区域，可见。

这样执行下来，虽然在事务A的执行期间，这一行数据被修改过，但是事务A不论在什么时候查，看到的数据结果都是一致的，所以我们称之为一致性读。



以上是从代码逻辑的角度来分析，整个过程稍微有点复杂。下面我们从时间先后顺序来分析，首先，我们需要知道的是，一个数据的版本，对于一个事务的视图来说，除了自己的更新总是可见以外，有以下三种情况：

- 版本未提交，不可见；
- 版本已提交，但是是在视图创建后提交的，不可见；
- 版本已提交，而且是在视图创建前提交的，可见。

而事务A中的查询语句是在事务A启动时生成的，此时：

- (1,3)还没有提交，属于情况1，不可见；
- (1,2)虽然提交了，但是是在视图数组创建之后提交的，属于情况2，不可见；
- (1,1)是在视图数组创建之前提交的，可见；



## 更新的逻辑

### 当前读

如果我们仔细观察，就会发现：事务B的update语句，如果按照一致性读，结果好像并不对。事务B的视图数组是在事务C提交之前创建的，此时它应该是看不见（1，2）的，怎么能算出（1，3）来呢？

![image-20200504215944927](/2020-05-03-MySQL-4/assets/images/image-20200504215944927.png)

是的，如果事务B在更新之前查询数据，返回的k的值确实是1。但是，当它要去更新数据的时候，就不能在历史版本上更新了，否则事务C的更新就丢失了。因此，事务B此时的set k=k+1是在（1，2）的基础上进行的操作。

而这里就用到了这样一条规则：更新数据都是先读后写的，而这个读，只能读当前的值，称为“当前读（current read）”。

因此，在更新的时候，当前读拿到的数据是（1，2），更新后生成了新版本的数据（1，3），这个新版本的row trx_id是101。

所以，事务B在执行查询语句时，一看自己的版本号是101，最新数据的版本号也是101，是自己的更新，可以直接使用，因此的得到的k值是3。



其实，除了update语句外，select语句如果加锁，也是当前读。因此，如果把事务A的查询语句`select * from t where id =1`修改一下， 加上lock in share mode或for update，也都可以读到版本号是101的数据，即返回的k的值是3。如下，就是这两个select语句，它们分别会加读锁（S锁，共享锁）和写锁（X锁，排它锁）：

```mysql
select k from t wehre id = 1 lock in share mode;
select k from t where id = 1 for update;
```



### 再看开篇的场景

这里，我们假设事务C不是马上提交的，而是变成了下面的事务C’：

![image-20200505101751839](/2020-05-03-MySQL-4/assets/images/image-20200505101751839.png)

事务C'的不同之处在于，更新后并没有马上提交，在它提交之前，事务B的更新语句先发起了。

在前面我们说过，虽然事务C'还没提交，但是（1，2）这个版本也已经生成了，并且是当前的最新版本。那么，事务B的更新语句会怎么处理呢？

这时，我们就不能忽略锁的存在了。事务C’没提交，也就是说（1，2）这个版本上的写锁还没释放。而事务B是当前读，必须要读最新版本，而且必须加锁，因此就被锁住了，必须等到事务C'释放这个锁，才能继续它的当前读。

![image-20200505102941779](/2020-05-03-MySQL-4/assets/images/image-20200505102941779.png)

这样一致性读、当前读和行锁就串联起来了。



### 再看开篇的问题

现在，我们再来看开篇的这个问题：事务的可重复读的能力是怎么实现的？



可重复读的核心就是一致性读（consistent read），而事务更新数据的时候，只能用当前读。如果当前记录的行锁被其它事务占用的话，就需要进入锁等待。



而读提交的逻辑和可重复读的逻辑类似，它们最主要的区别是：

- 在可重复读隔离级别下，只需要在事务开始时创建一致性视图，之后事务里的其它查询都共用这个一致性视图；
- 在读提交隔离级别下，每一个语句执行前都会重新算出一个新的视图。



如果隔离级别为读提交，那么事务A和事务B的查询语句查到的k，分别是多少呢？

![image-20200505110247043](/2020-05-03-MySQL-4/assets/images/image-20200505110247043.png)

我们可以看到，事务A的查询语句的视图数组是在执行这个语句的时候创建的，从时序上看，（1，2）、（1，3）的生成时间都在创建这个视图的时刻之前。但是，在这个时刻：

- （1，3）还没提交，因此，不可见；
- （1，2）已经提交了，因此，可见；

所以，这时事务A查询语句返回的是k=2，事务B查询语句返回的是k=3。
